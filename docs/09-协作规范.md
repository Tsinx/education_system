# 协作规范

## Git 分支建议

- `main`：稳定主分支。
- `feature/*`：新功能开发。
- `fix/*`：缺陷修复。
- `docs/*`：文档变更。

## 提交流程

1. 拉取或同步最新代码。
2. 在功能分支开发并自测。
3. 提交信息清晰说明变更目的。
4. 合并前完成代码 review 与关键流程回归。

## Commit 信息建议

建议使用：

- `feat:` 新功能
- `fix:` 修复
- `docs:` 文档
- `refactor:` 重构
- `chore:` 工程维护

示例：

- `feat: 新增知识图谱导出学习通模板接口`
- `fix: 修正导出层级整体后移一级`
- `docs: 补充项目文档组与 API 清单`

## 代码规范

- 前端：TypeScript 严格类型优先，接口类型先定义再使用。
- 后端：路由层轻逻辑，数据访问集中在 repository。
- 变更任何导入导出模板逻辑时，必须附带样例验证。

## 回归检查清单

- 课程创建/删除
- 资料上传与绑定
- 知识图谱加载
- 节点删除（递归/提升）
- 学习通模板导出
- 生成任务启动与结果导出

## 文档维护规则

- 代码接口变化时同步更新 `docs/06-API清单.md`。
- 知识图谱导入导出变化时同步更新 `docs/07-知识图谱导入导出规范.md`。
- 每次版本迭代在 `docs/CHANGELOG.md` 记录变更摘要。
